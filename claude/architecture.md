# iVelox Backend — Architecture & Structure

## Overview
Go REST API for iVelox IELTS learning platform. Follows Clean Architecture strictly.
Auth via Supabase JWT verification. DB is Supabase PostgreSQL (pgx driver).

## Tech Stack
| Layer | Technology |
|---|---|
| Language | Go 1.22+ |
| HTTP Framework | Gin |
| Database | Supabase PostgreSQL via pgx/v5 |
| Auth | Supabase JWT (HS256 verification) |
| AI Scoring | Gemini 2.0 Flash (writing + speaking) |
| Transcription | Groq Whisper (free tier, speaking) |
| Translation | DeepL API (auto-translate passages) |
| Config | godotenv |
| Docs | swaggo/swag (Swagger) |

## Architecture: Clean Architecture

```
Request
   │
   ▼
delivery/http      ← Gin handlers (parse request, call usecase, render JSON)
   │
   ▼
usecase            ← Business logic (imports domain interfaces only)
   │
   ▼
domain             ← Pure Go structs + Repository interfaces (zero external imports)
   ▲
   │
repository/postgres ← Implements domain interfaces (SQL queries, pgx)

infrastructure     ← Third-party clients (Supabase JWT, Gemini, Groq, DeepL)
middleware         ← Gin middleware (Auth JWT, CORS)
config             ← Env var loading
```

**Dependency rule:** Arrows point inward. Domain knows nothing about the outside world.

## API Routes

```
GET  /api/v1/health                         # Health check (no auth)

POST /api/v1/auth/verify                    # Verify JWT, return profile

GET  /api/v1/exams                          # List exams (filter: year, skill, source)
GET  /api/v1/exams/:id                      # Exam detail + sections
GET  /api/v1/exams/:id/sections/:sid        # Section content + translations

POST /api/v1/practice/sessions              # Start practice session
PATCH /api/v1/practice/sessions/:id         # Update session status
POST /api/v1/practice/answers               # Submit answer (reading/writing/listening)
POST /api/v1/practice/speaking              # Upload audio → Groq Whisper → Gemini score

GET  /api/v1/progress                       # User progress snapshots
GET  /api/v1/progress/recommendations       # Suggested next exams

GET  /api/v1/tips                           # Tips list (filter by skill)

# Admin only (role = 'admin' in profiles table)
POST /api/v1/admin/exams
POST /api/v1/admin/exams/:id/sections
PUT  /api/v1/admin/translations/:id
```

## Auth Flow
```
1. Frontend sends: Authorization: Bearer <supabase-jwt>
2. middleware/auth.go → infrastructure/supabase/jwt.go → VerifyJWT()
3. JWT valid → set c.Set("userID", claims.Sub)
4. Handler reads userID → calls usecase → repository queries DB
5. JWT invalid → 401 Unauthorized
```

## Error Response Format
Always return JSON:
```json
{"error": "human readable message"}
```

## Key Rules for Agents

1. **Never import gin/pgx in domain/ or usecase/** — domain is framework-free
2. **SQL only in repository/postgres/** — no raw SQL in handlers or usecases
3. **Handlers must be thin:** parse request → call usecase → c.JSON()
4. **All new routes** must be registered in `delivery/http/router.go`
5. **All new domain types** go in `internal/domain/`
6. **Tests use fakeRepo** (in-memory) — no real DB in unit tests
7. **Go binary:** `/opt/homebrew/bin/go` on macOS — always `export PATH="/opt/homebrew/bin:$PATH"`
8. **Before commit:** `go build ./...` + `go test ./...` must pass

## Database Schema

### User tables
| Table | Purpose |
|---|---|
| `profiles` | Basic info: display_name, email, avatar_url, provider, role |
| `user_goals` | Per-skill target band + date (set at onboarding, updatable) |
| `user_levels` | Current band per skill — seeded from onboarding quick test, upserted after sessions |
| `user_scores` | Append-only score history per session per skill (for charts) |
| `user_streaks` | Daily study streak + last_reminded_at (for push reminders) |

### Content tables (admin-managed)
| Table | Purpose |
|---|---|
| `exams` | Exam metadata (skill, difficulty, source) |
| `sections` | Passages / audio per exam |
| `questions` | MCQ, fill_blank, true_false, matching, short_answer |
| `translations` | Pre-computed DeepL translations per section per lang |
| `tips` | Study tips per skill and band range |

### Session tables (user-generated)
| Table | Purpose |
|---|---|
| `practice_sessions` | One session per exam attempt |
| `answers` | User answers with AI score/feedback |
| `progress_snapshots` | Legacy aggregate snapshots (superseded by user_scores) |

## Auth Flow
```
Email/password + Google OAuth → handled by Supabase Auth (client SDK)
Backend never receives raw passwords.

1. Frontend sends: Authorization: Bearer <supabase-jwt>
2. middleware/auth.go → infrastructure/supabase/jwt.go → VerifyJWT()
3. JWT valid → set c.Set("userID", claims.Sub)
4. Handler reads userID → calls usecase → repository queries DB
5. JWT invalid → 401 Unauthorized

On first login: trigger auto-creates profiles row.
Onboarding: user sets goals (user_goals) + takes quick test → user_levels seeded.
```
